# Frontend Enterprise UX Skill

按以下流程产出结果，保持“先业务目标，再信息架构，再交互细节”的顺序。

## 1) 明确业务语境

先向需求方确认或自行补齐以下信息：

1. 用户角色（例如：管理员、审核员、运维值班人）
2. 高频任务（每天/每周必须完成的动作）
3. 风险操作（删除、发布、批量变更、审批驳回）
4. 成功指标（完成时长、错误率、培训成本、工单量）

无法获取完整信息时，明确假设并在输出中标注“待验证假设”。

## 2) 套用企业 UX 设计原则

遵循以下原则进行方案筛选与取舍：

- **任务优先**：默认先满足“快、稳、少出错”，再追求视觉创新。
- **信息密度可控**：主界面允许较高密度，但必须提供分组、筛选、折叠与渐进披露。
- **一致性优先于局部最优**：按钮语义、表单校验、状态颜色、反馈文案在全局一致。
- **低认知切换成本**：列表页完成 80% 操作，详情页用于确认和深度编辑。
- **强反馈闭环**：提交、保存、异步任务、失败重试都要有清晰状态。

详细规范见 `references/enterprise-ux-patterns.md`。

## 3) 选择页面模式并组织结构

根据任务类型选用页面骨架：

- 台账/监控类：`筛选区 + 数据表格 + 右侧详情抽屉`
- 流程审批类：`待办列表 + 步骤轨迹 + 审批动作区`
- 主数据维护类：`树/分组导航 + 列表 + 表单编辑`

输出页面结构时，必须给出：

- 信息层级（L1/L2/L3）
- 默认展示字段与“可折叠”字段
- 首屏关键操作（不超过 3 个主操作）

## 4) 设计关键交互

对以下交互给出明确规则：

1. **查询与筛选**：默认筛选、清空、保存筛选方案、空结果提示。
2. **表格操作**：单选/多选、批量操作防误触、列宽与固定列策略。
3. **表单录入**：实时校验 vs 提交校验、必填提示时机、错误聚焦。
4. **危险动作**：二次确认文案、影响范围说明、可撤销策略。
5. **状态反馈**：加载、成功、失败、部分成功、超时。

## 5) 输出标准化交付物

按需求复杂度输出以下内容（至少包含前两项）：

- 页面 UX 说明（目标、用户、场景、流程）
- 信息架构草图（文本结构或线框描述）
- 交互规则清单（可直接交给前端开发）
- 文案与状态词典（按钮、提示、错误信息）
- 验收检查表（使用 `assets/review-checklist.md`）

## 6) 进行设计评审

评审时必须逐项检查：

- 是否降低关键任务的完成路径长度
- 是否减少错误操作概率
- 是否保证新手可学、老手高效
- 是否与既有组件和视觉语义保持一致
- 是否覆盖异常与边界状态

## 7) 产出格式要求

默认使用以下结构回复：

1. 业务目标与用户画像
2. 信息架构与页面分区
3. 核心任务流（步骤化）
4. 关键交互规则
5. 异常与边界处理
6. 验收清单

若用户要求“直接给页面方案”，先给“高保真文本线框”，再给交互细节，不省略状态与错误处理。
